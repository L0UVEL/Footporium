Footporium Database Structure Explanation

This document explains the structure of the `db_footporium` database, including its tables, columns, and relationships.


1. USERS TABLE (`users`)
Purpose: Stores all registered user accounts (both customers and admins).

Columns:
- `id`: Unique identifier for each user (Auto-incrementing).
- `full_name`: The user's full name.
- `email`: The user's email address (Must be unique).
- `phone`: Contact number (Optional).
- `password`: The user's password (Stored as a secure hash, not plain text).
- `role`: Defines permissions. Can be 'user' or 'admin'. Default is 'user'.
- `created_at`: Timestamp of when the account was created.

2. ADDRESSES TABLE (`addresses`)
Purpose: Stores shipping addresses. Separated from the users table to allow multiple addresses per user.

Columns:
- `id`: Unique identifier for each address.
- `user_id`: Links this address to a specific user in the `users` table.
- `address_line`: The actual address text.
- `city`: City name.
- `postal_code`: Zip/Postal code.
- `country`: Country name (Default: 'Philippines').
- `created_at`: Timestamp of creation.

Relationship:
- One User -> Many Addresses (1:N)
- `user_id` is a Foreign Key referencing `users.id`.

3. PRODUCTS TABLE (`products`)
Purpose: Stores the items available for sale.

Columns:
- `id`: Unique identifier for each product.
- `name`: Product name.
- `price`: Product price (Decimal format for accuracy).
- `image_url`: Path to the image file (e.g., 'assets/img/shoe.png').
- `description`: Detailed description of the product.
- `created_at`: Timestamp of creation.

4. ORDERS TABLE (`orders`)
Purpose: Stores the summary of a purchase (Header information).

Columns:
- `id`: Unique identifier for each order.
- `user_id`: The customer who placed the order.
- `address_id`: The specific address where this order should be shipped.
- `total_amount`: The total cost of the order.
- `status`: Tracks progress ('pending', 'processing', 'shipped', 'delivered', 'cancelled').
- `created_at`: Timestamp of creation.

Relationships:
- One User -> Many Orders (1:N)
- One Address -> Many Orders (1:N)

5. ORDER ITEMS TABLE (`order_items`)
Purpose: Stores the specific products inside an order (Line Items).

Columns:
- `id`: Unique identifier for each line item.
- `order_id`: Which order does this item belong to?
- `product_id`: Which product was bought?
- `quantity`: How many of this product?
- `price`: The price *at the time of purchase*.

Relationships:
- One Order -> Many Order Items (1:N)
- One Product -> Many Order Items (1:N)

6. PAYMENTS TABLE (`payments`)
Purpose: Stores payment details for an order.

Columns:
- `id`: Unique identifier for each payment.
- `order_id`: Links the payment to a specific order.
- `payment_method`: Method used ('cod', 'credit_card', 'gcash', 'maya').
- `amount`: Amount paid.
- `status`: Payment status ('pending', 'completed', 'failed').
- `transaction_id`: External reference ID (e.g., from GCash).
- `created_at`: Timestamp of creation.

Relationship:
- One Order -> One Payment (1:1 for this design)
